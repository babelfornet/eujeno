How it works

A model assembled from machines that never meet.

1

The model is sharded

A model's transformer blocks are split into contiguous layer ranges. Each range is a self-contained stage of the forward pass.

2

Nodes claim layers

Each peer hosts whatever layers it can fit in VRAM and announces them. The swarm self-organizes into a full end-to-end chain.

3

Anyone queries it

A request routes hop-by-hop through the chain — activations passed peer to peer — and the final node streams tokens back.

Why peer-to-peer

No data center. No single owner. No off switch.

⌗

Run models you can't host

A 70B model needs no 70B machine. Contribute the layers your GPU can fit and reach the whole thing through the swarm.

⟳

Self-healing routes

Nodes join and drop constantly. When a peer disappears, requests reroute through another holding the same layers. No downtime.

⚇

Permissionless

Anyone can join, host layers, or query the model. No accounts, no gatekeepers, no central API to revoke your access.

≋

Idle compute, reused

Built from spare consumer and prosumer GPUs — inference at a fraction of cloud cost, paid in contributed compute.

◇

One shared model

Every node points at the same weights. Upgrade once and the whole network serves the new model — no redeploy, no fork.

⊞

Scales sideways

More peers means more replicas per layer, more parallel requests, more throughput. Capacity grows with the crowd.
